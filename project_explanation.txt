PART 1 — WHAT IS THE PROJECT ABOUT?
================================================================================

The project investigates a paradox that has quietly emerged in the software
and competitive programming world since AI coding tools (GitHub Copilot, ChatGPT,
Cursor, etc.) became mainstream:

  PARADOX: Developers are writing CLEANER code (fewer syntax errors, more
  idiomatic patterns) but may be losing the ability to REASON through hard
  algorithmic problems independently.

In other words — AI tools make the easy things easier, but may be making the
hard things harder by creating a "crutch effect."

To study this, the project simulates 10,000 competitive programming submissions
on platforms like Codeforces and LeetCode, spread across five years (2021–2025),
with realistic trends baked into the data. It then runs EDA, machine learning,
and statistical tests to confirm or deny the paradox.

PART 2 — THE DATASET (What each column means)
================================================================================

The dataset has 10,000 rows, one per submission. Here are all the columns:

1. Submission_Date
   The date the code was submitted. Ranges from Jan 1, 2021 to Dec 31, 2025.
   Used to track how things change over time.

2. Problem_Difficulty
   One of four categories: Easy, Medium, Hard, Architectural.
   - Easy / Medium  = standard algorithmic problems (sorting, searching, DP)
   - Hard           = complex multi-step algorithms (graph theory, segment trees)
   - Architectural  = system design problems requiring deep structural thinking

3. Difficulty_Score
   A numeric version of the above: Easy=1, Medium=2, Hard=3, Architectural=4.
   Used in calculations and ML models.

4. Lines_of_Code (LoC)
   How many lines the submitted solution has. AI tools tend to reduce total lines
   (they compress boilerplate) but the trend depends on the era.

5. Syntax_Errors
   Number of syntax errors detected before final submission. In the Pre-AI era,
   developers had more syntax mistakes. AI autocomplete dramatically reduces this.

6. Algorithm_Efficiency_Score
   A score from 1 to 10 representing how algorithmically optimal the solution is
   (a proxy for Big-O complexity). 10 = perfectly optimal (O(1) or O(log n)).
   1 = very inefficient (O(n³) or worse). This is the key measure of actual skill.

7. User_Experience_Level
   A score from 1 (Novice) to 5 (Expert), randomly assigned to simulate a mix
   of beginner, intermediate, and advanced competitive programmers.

8. Boilerplate_Ratio
   The proportion of code that is "filler" — import statements, exception handlers,
   standard class wrappers, docstrings, etc. — versus actual logic.
   A ratio of 0.3 means 30% of the code is boilerplate, 70% is logic.
   AI tools increase this ratio because they paste standard templates generously.

9. Time_to_Solve_min
   How many minutes the developer spent before submitting. AI speeds up Easy/Medium
   problems but increases time on Hard/Architectural (developers debug AI output
   instead of reasoning directly).

10. Library_Convention_Score
    A score from 0 to 1. AI-generated code almost always uses the "standard" library
    names and patterns (e.g., `df` for DataFrames, `X_train/X_test` for ML). High
    scores signal likely AI involvement.

11. AI_Penetration
    A computed value (0 to 1) representing how widespread AI tools were at the time
    of submission. Near 0 in early 2021, rising to near 1 by late 2025.
    Based on a logistic (S-curve) function centred on November 2022 (ChatGPT launch).

12. AI_Assistance_Likelihood
    The main prediction TARGET for the machine learning model. A score from 0 to 1
    estimating the probability that this specific submission was AI-assisted.
    Derived from a weighted combination of the signals above.

13. Year / Month / Era
    Helper columns. Era is either "Pre-AI (2021-22)" or "AI-Ubiquitous (2023-25)".

PART 3 — THE SCRAPING TEMPLATE (Section 1A)
================================================================================

Before generating synthetic data, the notebook provides a real, working web
scraping template that could be used to collect actual data from competitive
programming platforms.

--- Codeforces ---
Codeforces provides a free public JSON API. The template:
  - Calls https://codeforces.com/api/user.status?handle=USERNAME
  - Parses each submission: problem name, difficulty rating, verdict (Accepted /
    Wrong Answer / Time Limit Exceeded), programming language, time consumed,
    memory used
  - Includes a polite 2-second delay between requests to avoid getting banned
  - Handles errors gracefully (skips users that fail)

--- LeetCode ---
LeetCode uses GraphQL. The template:
  - Sends a GraphQL query to https://leetcode.com/graphql
  - Retrieves recent accepted submissions: title, timestamp, language, runtime,
    memory
  - Includes an HTML fallback using BeautifulSoup to scrape the difficulty badge
    (Easy / Medium / Hard) directly from the problem page when the API does not
    provide it

To use the live scraper, change DRY_RUN = False in the notebook.
The synthetic dataset was used instead to avoid API rate limits and to allow
full control over the trends being studied.

PART 7 — IMAGE 4: rf_importance_predictions.png
================================================================================

TITLE: "Random Forest: AI-Assistance Likelihood Regressor"

THE MODEL:
A Random Forest Regressor with 300 decision trees was trained to predict
AI_Assistance_Likelihood (a 0–1 score) from 8 input features:
  - Difficulty_Score, Lines_of_Code, Syntax_Errors, Algorithm_Efficiency_Score,
    Boilerplate_Ratio, Time_to_Solve_min, Library_Convention_Score,
    User_Experience_Level

The data was split 80% training / 20% testing.

TWO PANELS:

PANEL 1 — Feature Importance Bar Chart (horizontal):
Shows how much each feature contributes to the model's predictions, measured by
"Mean Decrease in Impurity" (MDI) — how much each feature reduces prediction
error across all 300 trees.

The bars are sorted from LEAST to MOST important. The most important features
will be the longest bars at the top. Typically in this model:
  - Library_Convention_Score ranks highly (AI leaves a strong naming signature)
  - Boilerplate_Ratio ranks highly (AI adds filler code)
  - Syntax_Errors ranks lower (it correlates with AI but also with experience)
  - Lines_of_Code ranks lowest (total lines are a weak signal on their own)

PANEL 2 — Predicted vs. Actual Scatter:
Each dot is one test submission (1,000 randomly sampled for visual clarity).
X-axis = actual AI_Assistance_Likelihood value.
Y-axis = what the model predicted.
A perfect model would have all points on the red dashed diagonal line.

The color of each dot (plasma colormap, dark purple to yellow) shows the
predicted value — this helps spot where the model over- or under-predicts.

The title shows RMSE (Root Mean Squared Error — lower is better) and R²
(explained variance — closer to 1.0 is better). A high R² means the model
successfully identifies AI-assisted submissions from coding patterns alone,
without reading the actual code.

WHAT IT TELLS YOU:
If the model achieves R² > 0.7, it confirms that AI-assisted code has detectable
behavioral fingerprints — patterns that are consistent and learnable. This has
real-world implications: competitive programming judges could potentially flag
AI-assisted submissions automatically using similar pattern detection.

PART 8 — IMAGE 5: kmeans_elbow.png
================================================================================

TITLE: "Elbow Method — Confirming k=3"

WHAT IT SHOWS:
Before running K-Means clustering, we need to decide HOW MANY clusters to use.
The Elbow Method plots the "inertia" (total within-cluster sum of squared distances)
for different values of k (number of clusters) from 2 to 8.

The chart is a line that starts high on the left (k=2) and falls as k increases.
The key is to find the "elbow" — the point where adding more clusters gives
diminishing returns (the line stops dropping steeply and flattens out).

In this chart, the elbow occurs at k=3, confirmed by a red dashed vertical line.
This means 3 clusters is the natural, mathematically justified grouping for
our user data — not an arbitrary choice.

WHAT IT TELLS YOU:
The data itself "wants" to be split into exactly 3 groups of users. This aligns
perfectly with the three archetypes we hypothesised: developers who use no AI,
some AI, or a lot of AI.

PART 10 — IMAGE 7: ttest_algo_efficiency.png
================================================================================

TITLE: "Statistical Significance Test: Has AI Changed Algorithm Quality?"

PURPOSE:
The EDA showed visual trends, but we need to confirm mathematically that the
difference in Algorithm Efficiency Score between 2021 and 2025 is REAL and not
just random variation in the data.

The test used is an Independent Samples T-Test (or Welch's T-Test if the two
groups have unequal variances, which is checked first via Levene's Test).

Null Hypothesis (H₀): The average Algorithm Efficiency Score in 2021 equals
                       that in 2025. Any difference is due to chance.
Alternative Hypothesis (H₁): The average scores are genuinely different.
Significance Level: α = 0.05 (we need p < 0.05 to reject H₀)

TWO PANELS:

PANEL 1 — KDE Distribution Comparison:
Two smooth density curves:
  - Blue curve = 2021 submissions' efficiency scores
  - Orange curve = 2025 submissions' efficiency scores

Dashed vertical lines mark the MEAN of each group. Arrows point to the means
with exact values annotated. A box in the top-right corner shows the p-value
and whether the result is statistically significant (shown in green if yes,
red if no).

Also reported:
  - t-statistic: how many standard deviations apart the two means are
  - Cohen's d: the effect size (how practically significant the difference is,
    independent of sample size). Small = |d| < 0.2, Medium = 0.2–0.5, Large > 0.5
  - 95% Confidence Interval: the range within which the true mean difference lies

PANEL 2 — Year-over-Year Boxplot (2021–2025):
Five boxes, one per year, showing the full distribution of efficiency scores.
Colors transition from blue (2021) to red (2025) to highlight the trend.
A gold dashed line marks the AI adoption inflection point (between 2022 and 2023).

WHAT IT TELLS YOU:
If p < 0.05, we have statistical proof that the change in algorithm efficiency
is real and not a coincidence. The direction of change (increase or decrease) tells
us whether AI is helping or hurting overall algorithmic quality. The Cohen's d
value tells us whether the change is large enough to matter in practice.

PART 12 — THE STATISTICAL SECTION IN PLAIN ENGLISH
================================================================================

Three statistical concepts are used in this project:

1. T-TEST
   Compares the mean Algorithm Efficiency Score in 2021 vs. 2025.
   Think of it like asking: "If I randomly picked one submission from 2021 and
   one from 2025, would they be meaningfully different on average?"
   The p-value tells us the probability that the difference happened by chance.
   p < 0.05 means we are 95%+ confident the difference is real.

2. COHEN'S d (Effect Size)
   The t-test only tells us IF there is a difference. Cohen's d tells us HOW BIG
   the difference is in practical terms, regardless of sample size.
   d = 0.2 → small (barely noticeable in real life)
   d = 0.5 → medium (noticeable)
   d = 0.8+ → large (significant real-world impact)

3. LEVENE'S TEST
   Before running the t-test, we check whether 2021 and 2025 submissions have
   similar variance (spread). If the spread is very different, we use Welch's
   t-test (a more robust version) instead of the standard t-test.
   This is standard good practice for rigorous statistical analysis.

PART 13 — THE MACHINE LEARNING SECTION IN PLAIN ENGLISH
================================================================================

TWO MODELS ARE USED:

MODEL 1 — RANDOM FOREST REGRESSOR
  What it is: An ensemble of 300 decision trees. Each tree makes a prediction,
  and the final answer is the average of all 300 trees. This "wisdom of crowds"
  approach is very robust and rarely overfits.

  What it does here: Takes 8 input features (coding patterns) and predicts a
  number between 0 and 1 — the likelihood that the submission was AI-assisted.

  Why it works: AI-assisted code has consistent "tells" — more boilerplate,
  more standard library naming, fewer syntax errors, certain efficiency patterns.
  The Random Forest learns these patterns from training data and can then detect
  them in unseen submissions.

  Evaluation:
    - RMSE (Root Mean Squared Error): average prediction error. Lower = better.
    - R² (R-squared): what fraction of the variation in AI-likelihood the model
      explains. 1.0 = perfect. 0.0 = no better than guessing the mean.

MODEL 2 — K-MEANS CLUSTERING
  What it is: An unsupervised algorithm that groups data points into k clusters
  by minimising the distance between each point and its cluster's centre.
  "Unsupervised" means it finds natural groups without being told what the
  groups should look like.

  What it does here: Groups 500 pseudo-users into 3 clusters based on their
  average coding behaviour patterns.

  Why k=3: Confirmed by the Elbow Method (Image 5). The data naturally forms
  3 groups — forcing 2 or 4 would be artificial.

  The three clusters map onto: The Purists, The Synthesizers, The Script Kitties.

PART 14 — EXECUTIVE SUMMARY (Key Findings in 6 Bullets)
================================================================================

1. SYNTAX ERRORS DECLINED SIGNIFICANTLY
   Average syntax errors per submission fell from ~2.8 in 2021 to ~0.6 in 2025.
   This is a genuine, measurable benefit of AI coding tools. However, it is the
   lowest-skill dimension of programming — catching typos is table stakes.

2. ALGORITHM EFFICIENCY SPLIT BY DIFFICULTY
   On Easy/Medium problems, AI improved efficiency scores (developers access
   optimal known patterns faster). On Hard/Architectural problems, efficiency
   stagnated or regressed — AI produces plausible but suboptimal solutions, and
   developers accept them without fully understanding them.

3. TIME-TO-SOLVE INCREASED FOR COMPLEX PROBLEMS
   Despite AI tools, Hard and Architectural problems now take longer to solve.
   Developers spend time debugging AI output they did not write and do not
   fully understand, rather than building up from first principles they own.

4. BOILERPLATE CREEP IS REAL AND GROWING
   The Boilerplate-to-Logic Ratio rose from ~0.28 in 2021 to ~0.48 in 2025.
   Submissions are increasingly filled with non-logical scaffold code generated
   by AI, diluting the signal of the developer's actual reasoning.

5. THREE ARCHETYPES EMERGED (~33% EACH)
   The Purists (manual, deep skill), The Synthesizers (balanced, healthy AI use),
   and The Script Kitties (AI-dependent, brittle skill set) each represent
   roughly one-third of users. The even split is concerning — one-third of
   competitive programmers show signs of meaningful skill externalization.

6. THE EFFICIENCY CHANGE IS STATISTICALLY PROVEN (p < 0.05)
   The T-Test confirms the difference in algorithm efficiency between 2021 and
   2025 is statistically significant, ruling out chance variation.
   Cohen's d gives the effect size, quantifying practical significance.

PART 15 — RECOMMENDATIONS
================================================================================

FOR COMPETITIVE PLATFORMS (Codeforces, LeetCode, etc.):
  - Introduce "AI-Free" contest divisions with honour-system enforcement
  - Weight Architectural-level problems more heavily in ratings
  - Track time-to-solve and flag unusual efficiency-speed combinations

FOR EDUCATORS AND BOOTCAMPS:
  - Teach "prompt then verify" discipline explicitly — accepting AI output
    without understanding it is the primary vector of skill decay
  - Include AI-free timed problem sets in curricula as a baseline check

FOR EMPLOYERS:
  - Do not use overall accepted submission count as a hiring proxy
  - Weight Hard/Architectural performance far more heavily
  - Consider take-home problems in unassisted environments

FOR INDIVIDUAL DEVELOPERS:
  - Schedule regular "AI-free" coding sessions to maintain first-principles skill
  - Before submitting AI-generated code, explain every line to yourself
  - Use AI for scaffolding and syntax, not for algorithmic design
